# Створення HTML листа

Цей документ описує основні принципи створення HTML листів, які будуть коректно відображатися в різних поштових клієнтах.

## Структура

HTML листи мають свою специфічну структуру, яка відрізняється від звичайних веб-сторінок. Вони повинні бути простими і сумісними з різними поштовими клієнтами. Основна структура HTML листа включає:

```html
Content-Type: multipart/related; boundary="REL_BOUNDARY" 

--REL_BOUNDARY
Content-Type: multipart/alternative; boundary="ALT_BOUNDARY" 

--ALT_BOUNDARY
Content-Type: text/plain; charset="utf-8" 

(Plain‑text версія листа)

--ALT_BOUNDARY Content-Type: text/html; charset="utf-8" 

(HTML версія листа) 

--ALT_BOUNDARY--

--REL_BOUNDARY
Content-Type: image/png
Content-ID: <image1.png>
Content-Transfer-Encoding: base64

(base64…)

--REL_BOUNDARY
Content-Type: image/png
Content-ID: <image2.png>
Content-Transfer-Encoding: base64

(base64…)

--REL_BOUNDARY--
```

### Plain-text версія

Ця версія листа призначена для поштових клієнтів, які не підтримують HTML або для користувачів, які віддають перевагу текстовим листам. Вона повинна містити основну інформацію, представлену в HTML версії, але без форматування.

### HTML версія

HTML версія листа містить всі стилі, зображення та форматування. Вона повинна бути розроблена з урахуванням сумісності з різними поштовими клієнтами, тому слід уникати складних CSS властивостей і використовувати інлайн-стилі.

Робота з зображеннями є дуже спеціфічною для HTML листів, оскільки деякі поштові клієнти можуть блокувати зовнішні зображення або не підтримувати певні формати. Тому важливо правильно включати зображення в листи, щоб забезпечити їх коректне відображення.

**Рекомендується викорситовувати CID (Content-ID) для вбудованих зображень**, що дозволяє забезпечити їх відображення навіть при блокуванні зовнішніх зображень. Зовнішні посилання на зображення можуть бути використані, але вони можуть бути заблоковані деякими поштовими клієнтами, тому їх використання слід обмежувати.

## Зображення

Зображення в HTML листах можуть бути включені двома способами: як вбудовані (inline) або як зовнішні посилання. Вбудовані зображення використовують `Content-ID` для посилання на них у HTML коді, тоді як зовнішні посилання вказують на URL зображення.

### Вбудовані зображення

```html
<img src="cid:image1.png" alt="Image 1">
```

Вбудовані зображення забезпечують кращу сумісність, оскільки вони включені безпосередньо в лист і не залежать від зовнішніх ресурсів. Однак вони можуть збільшувати розмір листа, що може вплинути на швидкість завантаження.

Вбудовані зображення розташовуються в секції `multipart/related` і мають унікальний `Content-ID`, який використовується для посилання на них у HTML коді.

Кожне вбудоване зображення має бути описано відповідно до наступного формату:

```
--REL_BOUNDARY
Content-Type: image/png
Content-ID: <image1.png>
Content-Transfer-Encoding: base64

(base64…)
```

### Зовнішні посилання

```html
<img src="https://example.com/image1.png" alt="Image 1">
```

Зовнішні посилання можуть бути проблематичними, оскільки деякі поштові клієнти блокують їх за замовчуванням. Вбудовані зображення забезпечують кращу сумісність, але збільшують розмір листа.

Посилання на зображення повинні бути захищеними (https://...) та вказувати на прямий графічний файл.

> [!NOTE]
> Використовуйте PNG формат, він підтримується більшістю поштових клієнтів і забезпечує хорошу якість зображення. **Уникайте використання SVG, оскільки він не підтримується всіма клієнтами.**

**Важливо!** Завжди додавайте атрибут `alt` до зображень, щоб забезпечити доступність і зрозумілість вмісту для користувачів, які не можуть бачити зображення.

Переконайтеся, що домен відправника має налаштовані `SPF` та `DKIM` записи, щоб уникнути потрапляння в спам.

> [!NOTE]
> **SPF** - це механізм, який дозволяє власникам доменів визначати, які сервери мають право відправляти електронну пошту від їх імені.
>
>
> **DKIM** - це метод автентифікації електронної пошти, який використовує криптографічні підписи для підтвердження того, що лист не був змінений під час передачі. Він допомагає забезпечити цілісність повідомлення і підтвердити його походження.

## Стилі

Стилі в HTML листах повинні бути простими і використовуватися з обережністю. Більшість поштових клієнтів підтримують лише базові CSS властивості, тому слід використовувати лише інлайн-стилі для забезпечення сумісності.

```html
<div style="background-color: #f0f0f0; padding: 20px;">
  <h1 style="color: #333;">Заголовок</h1>
  <p style="color: #666;">Текст листа...</p>
</div>
```

### Тег `<style>`
Деякі поштові клієнти підтримують використання тегу `<style>`, але це не гарантується. Якщо ви використовуєте цей тег, переконайтеся, що він розміщений у `<head>` секції листа і містить лише базові стилі.

```html
<head>
  <style>
    .header {
      background-color: #f0f0f0;
      padding: 20px;
    }
    .header h1 {
      color: #333;
    }
    .content p {
      color: #666;
    }
  </style>
</head>
<body>

</body>
```

### Завантаження шрифтів

Завантаження зовнішніх шрифтів у HTML листах не рекомендується, оскільки багато поштових клієнтів блокують їх за замовчуванням. Якщо вам потрібно використовувати спеціальні шрифти, розгляньте можливість використання зображень для заголовків або важливих текстових елементів. Якщо ви все ж хочете використовувати веб-шрифти, додавайте їх підтримку в `<style>` секції листа та завжди додавайте fall-back шрифти. Використовуйте лише WOFF2 формат, для забезпечення сумісності з більшістю поштових клієнтів.

```html
<style>
  @font-face {
    font-family: 'CustomFont';
    src: url('https://example.com/fonts/customfont.woff2') format('woff2');
  }
  .custom-text {
    font-family: 'CustomFont', Arial, sans-serif;
  }
</style>
```

## Тестування

Після створення HTML листа важливо протестувати його в різних поштових клієнтах, щоб переконатися, що він відображається правильно. Використовуйте інструменти для тестування електронної пошти, такі як Litmus або Email on Acid, щоб перевірити сумісність і вигляд вашого листа на різних платформах.

### Тестування в Outlook

Створіть лист, який ви хочете або плануєте розіслати клієнтам. Файл має мати розширення `.eml`. На початку листа додайте відповідні заголовки. Щоб перевірити заповнення відповідних полів в Outlook при перегляді листа.

```
From: support@raiffeisen.ua
To: recipient@example.com
Subject: =?UTF-8?B?Base64...==?=
MIME-Version: 1.0
Content-Type: text/html; charset=UTF-8
Content-Transfer-Encoding: 8bit
```

> [!NOTE]
> Використовуйте для **Subject** кодування `Base64`, щоб забезпечити правильне відображення символів у темі листа. Це особливо важливо для підтримки української мови та інших спеціальних символів.
